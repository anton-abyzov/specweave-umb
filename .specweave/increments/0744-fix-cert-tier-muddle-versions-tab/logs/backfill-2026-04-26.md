## Step 3 — `POST /api/v1/admin/trusted-author-backfill` (existing endpoint) — FAILED SAFELY

**Purpose:** flip `Skill.certTier` (parent record, not per-version) from `VERIFIED` to `CERTIFIED` for vendor-org skills, so the skill detail-page header badge displays correctly.

**Result:** `HTTP 200`, time = 15.5s, but functionally a no-op:
```json
{ "ok": true, "updated": 0, "errors": 151, "total": 151,
  "taintCleared": 0, "failures": [...151 entries...] }
```

**Failure breakdown:**
- 141× `Error: DB circuit open — skipping Skill.findUniqueOrThrow`
- 10× `Error: DB query timed out after 15000ms [Submission.findFirst]`

**Diagnosis:** the existing `trusted-author-backfill` endpoint calls `updateSkillTrust()` per row, which itself does ~7-9 sub-queries (Skill, Submission, ScanResult, BlocklistEntry, etc.). At 151 candidate rows × ~9 queries = 1300+ queries against the worker-bound DB pool. The first 10 timed out at 15s, opening the DB circuit breaker, which then short-circuited the remaining 141. **Zero rows were partially updated** — the failure is clean.

**Decision:** halted Step 3 here. Pushing further state changes against a degraded DB would be unsafe. The `Skill.certTier` mismatch (vendor-org skills still showing `VERIFIED` on the header badge) remains. This is a pre-existing issue unrelated to the 0744 scanner.ts bug — `trusted-author-backfill` was already in this state before 0744.

## Recommended follow-up increment

1. **Fix `trusted-author-backfill` scaling** — replace per-row `updateSkillTrust()` with a single bulk `UPDATE Skill SET certTier='CERTIFIED', certMethod='VENDOR_AUTO', trustScore=100, trustTier='T4' WHERE author IN (vendor_orgs) AND certTier!='CERTIFIED'`. Vendor authors get auto-CERTIFIED unconditionally — no per-row trust recomputation needed.
2. **Audit other `certTier` write sites** — at minimum:
   - `src/app/api/v1/admin/blocklist/[id]/unblock/route.ts:52` does `data: { certTier: "VERIFIED" }` unconditionally; same class of bug as the scanner.ts hardcode.
   - `src/app/api/v1/admin/rebuild-index/route.ts` and `eval/reverify/route.ts` reference `certTier: "VERIFIED"` in queries — verify those are filters, not writes.
3. **Add structured server logging** to `recompute-version-cert-tiers` (per-row log line including `before`/`after` so the response is reconstructible from CF tail logs).
4. **Architecture invariant test** — fail builds if any `.ts` file under `src/` writes `certTier:` to a Prisma `data` block without going through `deriveCertTier()`. Prevents the scanner.ts bug class from recurring.
