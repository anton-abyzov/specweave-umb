### C1: Excalidraw Source Files (`static/img/skills/diagrams/`)

Each `.excalidraw` file is valid Excalidraw JSON containing the editable diagram. These are the authoritative sources; SVGs are derived artifacts.

**Why a `diagrams/` subfolder?** Keeps editable sources separate from production assets. The `static/img/skills/` directory holds only SVGs that Docusaurus serves directly. This follows the existing pattern where `static/img/` holds only production-ready images (`.svg`, `.png`, `.jpg`), not source files.

**Naming convention**: Kebab-case matching the user story slug. The `.excalidraw` extension (not `.excalidraw.json`) is the Excalidraw community standard and what the VS Code extension recognizes.

### C2: SVG Exports (`static/img/skills/`)

Flat SVG files at `static/img/skills/` level. Each SVG corresponds 1:1 to an Excalidraw source. SVGs are committed to the repo so the docs build does not depend on Excalidraw tooling.

**Docusaurus path resolution**: From `docs/skills/why-skills-matter.md`, SVGs are referenced via the Docusaurus static asset convention:

```markdown
![What is a Skill? Before and after comparison](/img/skills/what-is-a-skill.svg)
```

This uses the `/img/` root because Docusaurus maps `static/` to the site root. Relative paths (`../../static/img/...`) should not be used -- Docusaurus recommends absolute paths from the static root.

### C3: "Why Skills Matter" Doc Page (`docs/skills/why-skills-matter.md`)

**Sidebar integration**: The `skillsSidebar` in `sidebars.ts` uses explicit `type: 'doc'` entries (not `autogenerated`). The new page must be added as an explicit entry.

**Placement decision**: Insert `why-skills-matter` as the SECOND item in `skillsSidebar`, between `skills/index` (Overview) and `skills/fundamentals` (Skills, Plugins & Marketplaces). ## Diagram Design System

All 4 diagrams must share a consistent visual language. This section defines the design system.

### Color Palette

| Role | Hex | Usage |
|------|-----|-------|
| Bad path / "Without" | `#FF6B6B` | Backgrounds, borders, arrows on "before" side |
| Bad path text | `#C92A2A` | Labels on red elements |
| Good path / "With" | `#51CF66` | Backgrounds, borders, arrows on "after" side |
| Good path text | `#2B8A3E` | Labels on green elements |
| Neutral / structural | `#868E96` | Dividers, neutral labels, prompt boxes |
| Background | `#FFFFFF` | Canvas background |
| Text (primary) | `#343A40` | Main labels and annotations |

These colors are from the Excalidraw default palette (which aligns with Open Color), ensuring they are available without custom configuration.

### Typography

- **Font**: Excalidraw hand-drawn font (Virgil) -- the default. Do not switch to "Normal" or code fonts.
- **Title labels**: Large size (28-36px equivalent in Excalidraw)
- **Section labels**: Medium size (20-24px)
- **Annotation text**: Small size (16px)
- **All text must be readable at 50% scale** (AC-US1-04): minimum effective size at 50% is 8px, so source text should be >= 16px.

### Layout Pattern

Every diagram follows the same left-right split:

```
+-----------+-----+------------+
|           |     |            |
|  WITHOUT  |  |  |   WITH    |
|  (red)    |  |  |  (green)  |
|           |  |  |            |
+-----------+-----+------------+
    "Before"  divider  "After"
```

- **Canvas width**: ~1200px (renders well in docs at various widths)
- **Canvas height**: ~600px (fits comfortably in a docs page without excessive scrolling)
- **Divider**: Vertical dashed line in neutral gray
- **Section headers**: "Without Skills" (left, red) and "With Skills" (right, green) at top of each half
- **Annotation arrows**: Hand-drawn style, same color as their section

### Per-Diagram Content Specifications

**Diagram 1 -- "What is a Skill?"**
- Left (red): Generic prompt box --> "Raw LLM" --> "Generic Output" (vague, boilerplate)
- Right (green): Same prompt box --> "LLM + Skill" --> "Production-Grade Output" (specific, structured)
- Key annotation: Arrow pointing to skill component with label "Domain knowledge injected here"

**Diagram 2 -- "How Skills Work"**
- Left (red): Prompt --> LLM (plain box) --> Output with "?" quality markers
- Right (green): Prompt --> LLM (box augmented with "Skill Knowledge" side-panel) --> Output with checkmark quality markers
- Key annotation: The skill knowledge panel shows abstract labels: "Patterns", "Rules", "Examples"

**Diagram 3 -- "Creating Skills"**
- Left (red): Scattered text bubbles ("use TypeScript", "handle errors", "add tests") -- ad-hoc, inconsistent
- Right (green): Structured SKILL.md anatomy: "Frontmatter" block, "Instructions Body" block, "Supporting Files" block -- labeled sections

## SVG Export Workflow

### Process

1. **Create/edit** in Excalidraw (VS Code extension or excalidraw.com)
2. **Export to SVG** using Excalidraw's built-in export:
   - Export with background: ON (white background for consistent rendering)
   - Embed scene: OFF (keeps SVG size small; the `.excalidraw` source is the editable version)
   - Scale: 1x
   - Padding: 10px (default)
3. **Save SVG** to `static/img/skills/{name}.svg`
4. **Verify** SVG renders in browser and in Docusaurus local dev (`npm start`)

### Export Settings Rationale

- **White background ON**: Without it, SVGs inherit the page background, which breaks in dark mode or colored contexts. White background ensures consistent rendering everywhere (docs, presentations, blog posts).
- **Embed scene OFF**: Embedding the scene data doubles the SVG file size. Since the `.excalidraw` source files are committed alongside, embedding is redundant.
- **1x scale**: Higher scales increase file size without benefit -- SVGs are vector and scale infinitely. 1x keeps files small (typically 20-80KB per diagram).

### Naming Convention

Source and export share the same base name:
- `diagrams/what-is-a-skill.excalidraw` --> `what-is-a-skill.svg`
- `diagrams/how-skills-work.excalidraw` --> `how-skills-work.svg`
- `diagrams/creating-skills.excalidraw` --> `creating-skills.svg`
- `diagrams/skill-eval-testing.excalidraw` --> `skill-eval-testing.svg`

## Technical Challenges

### Challenge 1: Excalidraw JSON Generation by AI

**Problem**: Claude cannot run Excalidraw's GUI. The `.excalidraw` files must be authored as raw JSON.

**Solution**: Excalidraw files are well-documented JSON with a known schema. Each file contains `elements` (rectangles, text, arrows, lines) with explicit coordinates, colors, and dimensions. The JSON can be authored directly and validated by opening in Excalidraw. The design system above provides exact colors and layout constraints to ensure consistency across all 4 files.

**Risk**: Coordinates may need manual adjustment after initial generation.
**Mitigation**: Keep elements on a simple grid (multiples of 20px). Use the Excalidraw VS Code extension to visually verify and nudge elements.

### Challenge 2: SVG Export Without GUI

**Problem**: Excalidraw's SVG export is typically done via the GUI. Programmatic export requires `@excalidraw/utils` npm package.

**Solution**: Two options, in order of preference:
1. **Manual export**: Open each `.excalidraw` file in the VS Code extension or excalidraw.com, export as SVG. This is a one-time operation for 4 files.
2. **Programmatic export**: Use `npx @excalidraw/utils` or a small Node script if automation is needed. Not worth the setup cost for 4 files.

**Recommendation**: Manual export. Document the export settings (background on, embed scene off, 1x scale) in the diagram design system above so future contributors can reproduce.

### Challenge 3: Dark Mode Compatibility

**Problem**: Docusaurus supports dark mode. SVGs with white backgrounds will look out of place.

**Solution**: Export with white background for now (matches existing SVGs like `workflow-diagram.svg` and `specweave-flow.svg` which also have white/light backgrounds). Dark-mode-adaptive SVGs require CSS-in-SVG tricks that are out of scope for this increment. If dark mode support is needed later, the `.excalidraw` sources can be re-exported with transparent backgrounds and CSS `filter: invert()` applied via Docusaurus theme customization.
